Lesson 12 · 模型接入

自定义 LLM Provider：接入你自己的模型

时长约 12 分钟

第 9–11 课你把 Extension 的四种注册方式学了三样。现在回到 Mission 的核心目标之一： 自定义模型供给。用 pi.registerProvider 接入你喜欢的 LLM API。

本课目标 到这一课结束时，你应该能：写出一个完整的 Provider Extension，接入任何 OpenAI 兼容 API（DeepSeek、智谱、OpenRouter、Ollama 等），并理解每个配置字段的含义。

Provider 是什么

Provider 是一个注册到 Pi 的模型源。它告诉 Pi：这里有一个 API 端点，上面挂了这些模型，调用方式是 xxx。 注册后，Pi 的模型选择器里就会出现你注册的模型。

调用签名是两个参数——provider 名 + 配置对象：

pi.registerProvider("my-llm", {
  baseUrl: "https://api.example.com/v1",
  apiKey: "$MY_API_KEY",
  api: "openai-completions",
  models: [
    {
      id: "my-model-a",
      name: "Model A",
      reasoning: false,
      input: ["text"],
      cost: { input: 0, output: 0, cacheRead: 0, cacheWrite: 0 },
      contextWindow: 128000,
      maxTokens: 4096,
    },
  ],
});
易踩的坑 第一个参数是 provider 的 ID（小写、不可重复），第二个才是配置。 如果写成 registerProvider({ name: "...", ... }) 单参数形式，Pi 会把整个对象当成 ID、把 undefined 当成配置，启动时报 Cannot read properties of undefined (reading 'streamSimple')。

字段详解：Provider 级

第二个参数 config 对象支持这些字段：

字段必填说明
baseUrl 定义 models 时必填 API 端点地址（OpenAI 兼容通常是 https://host/v1，Pi 会自动拼 /chat/completions）
apiKey 定义 models 时必填
（除非用 oauth）		 认证密钥。支持字面量、$ENV / ${ENV} 环境变量插值、!cmd 执行命令
api 定义 models 时必填 API 协议类型，决定 Pi 怎么发请求/解析响应。最常用："openai-completions"（标准 Chat Completions）
name 可选 给用户看的显示名（不是 ID）。不填则显示 provider ID
headers 可选 附加到每个请求的自定义 HTTP 头
authHeader 可选 true 时自动添加 Authorization: Bearer <apiKey>。大多数 OpenAI 兼容 API 默认就会加，不用设
models 可选 该端点提供的模型列表。只要填了，就会替换该 provider 的全部模型
oauth 可选 OAuth 登录流（用于 GitHub Copilot 这类需要 OAuth 的服务）。本课不讲
streamSimple 可选 完全自定义流式请求逻辑，用于非标准 API。本课末尾会提一下
Provider ID 的语义 第一个参数（provider ID）如果和内置重名（如 "anthropic"、"openai"、"deepseek"），就是覆盖：替换/修改内置 provider 的配置。 如果是新名字（如 "my-llm"），就是新增：和内置 provider 并存。 想给内置模型套一个代理？直接 registerProvider("anthropic", { baseUrl: "..." }) 即可。

字段详解：Model 级

models 数组里每个对象的字段：

字段必填说明
id 是 调 API 时实际传的 model 字段值（如 "deepseek-chat"）
name 是 UI 显示名（如 "DeepSeek V3"）
reasoning 是 是否支持扩展思考（DeepSeek-R1、Claude thinking、o1 这类）。决定 Pi 是否给模型发思考级别参数
input 是 支持的输入类型数组：["text"] 或 ["text", "image"]。决定能否贴图
cost 是 每百万 token 的费用（美元）：{ input, output, cacheRead, cacheWrite }。用于 Pi 统计花费，不知道就全填 0
contextWindow 是 模型上下文窗口大小（输入 token 上限）。DeepSeek V4 是 1000000，GPT-4o 是 128000
maxTokens 是 模型单次输出的最大 token 数。和 contextWindow 不是一回事
api 可选 覆盖 provider 级的 api。同一个 provider 下混搭不同协议时用
baseUrl 可选 覆盖 provider 级的 baseUrl
thinkingLevelMap 可选 把 Pi 的思考级别（minimal/low/medium/high/xhigh）映射到 API 实际接受的值。null 表示该级别不支持
compat 可选 OpenAI 兼容性微调（见下文）
headers 可选 只对这个 model 生效的请求头

关于 api 类型

Pi 内置支持的协议（最常用三个加粗）：

  • "openai-completions" —— OpenAI Chat Completions（/v1/chat/completions）。DeepSeek、智谱、Moonshot、通义、OpenRouter、Ollama 等绝大多数国产/开源 API 都用这个
  • "openai-responses" —— OpenAI Responses API（新版 /v1/responses）。官方 OpenAI 新接口
  • "anthropic-messages" —— Anthropic Messages API。Claude 官方 + 一些代理
  • "azure-openai-responses" / "openai-codex-responses" / "google-generative-ai" / "google-vertex" / "mistral-conversations" / "bedrock-converse-stream" —— 各家专用协议

如果接的服务说"OpenAI 兼容"，99% 是 "openai-completions"。

关于 compat

不同厂商的"OpenAI 兼容"实现细节有差异，compat 用来告诉 Pi 这些差异：

字段说明
supportsStore是否支持 OpenAI 的 store 参数（默认 true，多数国产 API 要设 false）
supportsDeveloperRole是否支持 developer role（OpenAI 新版），不支持就 false
requiresReasoningContentOnAssistantMessages思考模型是否要求 assistant 消息里带上 reasoning_content 字段（DeepSeek 要 true）
thinkingFormat思考内容的序列化格式，"deepseek" / "anthropic" / "openai" 等
forceAdaptiveThinkingAnthropic 兼容且要求 adaptive thinking 时设 true

如果不知道填什么，先留空（不设 compat）；遇到思考模型输出错乱、或参数不识别的报错，再回头按官方内置 provider 的同款配置抄。

动手：接入 DeepSeek

建 ~/.pi/agent/extensions/provider-deepseek.ts。 注意：Pi 内置已经有 deepseek provider（直连 api.deepseek.com），如果你要走自己的网关或代理，建议换一个 ID 避免覆盖。这里用 "my-deepseek"： 

import type { ExtensionAPI } from "@earendil-works/pi-coding-agent";

export default function (pi: ExtensionAPI) {
  pi.registerProvider("my-deepseek", {
    name: "DeepSeek (自定义)",
    baseUrl: "https://api.deepseek.com/v1",
    apiKey: "$DEEPSEEK_API_KEY",
    api: "openai-completions",
    models: [
      {
        id: "deepseek-chat",
        name: "DeepSeek V3",
        reasoning: false,
        input: ["text"],
        cost: { input: 0.27, output: 1.1, cacheRead: 0.07, cacheWrite: 0 },
        contextWindow: 64000,
        maxTokens: 8192,
      },
      {
        id: "deepseek-reasoner",
        name: "DeepSeek R1",
        reasoning: true,
        input: ["text"],
        cost: { input: 0.55, output: 2.19, cacheRead: 0.14, cacheWrite: 0 },
        contextWindow: 64000,
        maxTokens: 8192,
        compat: {
          supportsStore: false,
          supportsDeveloperRole: false,
          requiresReasoningContentOnAssistantMessages: true,
          thinkingFormat: "deepseek",
        },
      },
    ],
  });
}

启动前：export DEEPSEEK_API_KEY=sk-xxx。 重启 Pi，用 /model（或 /provider）就能看到 my-deepseek/deepseek-chat 出现在列表里。

两种写法选一

如果你只是想把内置 DeepSeek 走的地址改到自己代理，不重新声明模型，那就直接覆盖： pi.registerProvider("deepseek", { baseUrl: "https://your-proxy/v1" })——只改 baseUrl，模型列表自动保留。

如果代理上挂的是不同的模型，就得像上面那样声明 models，并且 provider ID 要换一个（覆盖内置时，模型列表会被完全替换）。

动手试试

  1. 创建一个 Provider Extension 文件。
  2. 设置对应 API Key 的环境变量。
  3. 重启 Pi，输入 /model 确认新模型出现在列表里。
  4. 用 /model my-deepseek/deepseek-chat（或屏幕上提示的格式）切换到新模型。
  5. 输入任意一句话，确认回复来自你的模型。

动手：接入本地 Ollama

本机跑了 Ollama 的话同样一个 Extension 就能接入：

import type { ExtensionAPI } from "@earendil-works/pi-coding-agent";

export default function (pi: ExtensionAPI) {
  pi.registerProvider("ollama", {
    name: "Ollama (Local)",
    baseUrl: "http://localhost:11434/v1",
    apiKey: "ollama", // Ollama 不需要密钥，但字段不能为空
    api: "openai-completions",
    models: [
      {
        id: "llama3.2",
        name: "Llama 3.2 (8B)",
        reasoning: false,
        input: ["text"],
        cost: { input: 0, output: 0, cacheRead: 0, cacheWrite: 0 },
        contextWindow: 8192,
        maxTokens: 4096,
      },
      {
        id: "qwen2.5-coder",
        name: "Qwen 2.5 Coder",
        reasoning: false,
        input: ["text"],
        cost: { input: 0, output: 0, cacheRead: 0, cacheWrite: 0 },
        contextWindow: 8192,
        maxTokens: 4096,
      },
    ],
  });
}
为什么本地模型还要填 cost Pi 用 cost 统计会话花费；本地模型不扣钱，但字段必填，全置 0 即可。 apiKey 也类似——Ollama 不验签，但 Pi 的 schema 要求非空字符串。

高级：自定义请求头与流式逻辑

真实场景里你可能需要：

// 1. 走企业代理时加自定义鉴权头
pi.registerProvider("corp-ai", {
  baseUrl: "https://ai.corp.com/v1",
  api: "openai-completions",
  headers: {
    "X-Corp-Auth": "$CORP_AUTH_TOKEN", // 同样支持 $ENV 插值
  },
  models: [/* ... */],
});

// 2. 动态发现模型（async 工厂函数）
export default async function (pi: ExtensionAPI) {
  const res = await fetch("http://localhost:1234/v1/models");
  const data = (await res.json()).data;
  pi.registerProvider("local-lm-studio", {
    baseUrl: "http://localhost:1234/v1",
    apiKey: "lm-studio",
    api: "openai-completions",
    models: data.map((m) => ({
      id: m.id,
      name: m.id,
      reasoning: false,
      input: ["text"],
      cost: { input: 0, output: 0, cacheRead: 0, cacheWrite: 0 },
      contextWindow: 8192,
      maxTokens: 4096,
    })),
  });
}

对于协议完全不标准的 API（既不是 OpenAI 也不是 Anthropic 格式），还有 streamSimple 这个终极扩展点——你自己实现流式收发。 这是高级主题，需要时再翻官方 docs/custom-provider.md 的 Custom Streaming 章节。
