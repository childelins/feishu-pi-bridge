Lesson 13 · 扩展方式

Skills 与 Prompt Templates：教 LLM 做事的配方

时长约 9 分钟

第 8 课讲过 Pi 有三种扩展方式：Extension、Skill、Prompt Template。 前 12 课你一直在写 Extension（TypeScript 代码），这一课学不用写代码的两种方式。

本课目标 到这一课结束时，你应该能：写一个 Skill Markdown 文件让 LLM 自动加载， 写一个 Prompt Template 复用提问结构，以及判断什么场景该用 Skill 而非 Extension。

Skill 是什么

Skill 是一个 Markdown 文件，注入到系统提示词中。它教 LLM 怎么做一件事——格式规范、工作流程、注意事项——而不是"帮用户执行代码"。 LLM 会在每次对话时自动读取匹配的 Skill 文件。

ExtensionSkill
本质 TypeScript 代码 Markdown 文档
谁执行 Node.js 运行时 LLM 阅读并遵循
适合 交互、调 API、读写文件 编码规范、工作流、行为约束
修改后 需要重启 Pi 只需要 /reset 清空上下文

写第一个 Skill

Pi 会在几个目录中自动发现 Skill 文件，文件名即有效名。 建 ~/.pi/agent/skills/python-style.md： 

按照公司 Python 编码规范进行所有修改：

## 格式
- 每行不超过 88 个字符
- 使用双引号而非单引号
- 类型注解为必选项
- 结尾留一个空行

## 导入顺序
1. 标准库
2. 第三方库（空行分隔）
3. 本地模块（空行分隔）

## 例外
如果你被要求修改一个不符合此规范的文件，保留原有的引号和缩进风格。

保存后启动 Pi，输入一条和 Python 相关的指令，LLM 会自动读取这个 Skill。 你会注意到它生成的代码自动遵循上面的规范。

Skill 是怎么加载的？ Pi 在每次对话开始时，在 ~/.pi/agent/skills/ 和 .pi/skills/ 中扫描 .md 文件， 把文件名（不含后缀）和内容一起注入系统提示词：[skill: python-style]。 文件名就是 Skill 名，LLM 按需使用它们。

Skill 的生效范围

路径范围
~/.pi/agent/skills/ 全局——所有项目都生效
.pi/skills/ 项目本地——仅当前项目生效（和 .pi/extensions/ 同理）
~/.pi/agent/skills/<project-dir>/ 目录绑定——仅在特定项目目录下生效（例如 ~/.pi/agent/skills/my-project/ 只在 /path/to/my-project 中加载）

覆盖优先级：项目本地 < 目录绑定 < 全局。如果同一个文件名出现在多个位置，最具体的那个生效。

动手：项目级 Skill

动手试试

  1. 创建 ~/.pi/agent/skills/python-style.md（用上面的内容）。
  2. 重启 Pi，问"写一个读取 JSON 文件的 Python 函数"。
  3. 看 LLM 的输出是否用双引号、有类型注解、< 88 字符。
  4. 试试 ~/.pi/agent/skills/teach-pi/ 目录绑定的方式，只在本项目生效。

自动加载的 Skill

Pi 在启动时还会自动加载一些由安装在项目中的 npm 包提供的 Skill。 这些包以 @pi/skill- 或 pi-skill- 前缀命名，通过 pi install 安装。 

pi install @pi/skill-git-commit

安装后，Pi 会在合适的时候自动注入对应 Skill 的内容——不需要用户手动引用。

Skill vs Context File Context File（第 6 课）是你手动用 /context 指定的文件。 Skill 是 Pi 自动扫描和注入的——你不需要主动引用它。

Prompt Templates

Prompt Template 是另一种不用写代码的扩展方式。 它是一段带变量的文本模板，展开为完整提示词供 LLM 使用。 适合你频繁使用但每次参数不同的提问模式。

Prompt Template 放在 ~/.pi/agent/prompts/ 或 .pi/prompts/ 中， 格式是 .md 文件，内容用 --- 分隔 frontmatter 和正文： 

---
name: "review-pr"
description: "Review a pull request and suggest improvements"
---

请审查以下 Pull Request 修改：

代码变更：{{diff}}

请重点关注：
1. 是否有逻辑错误
2. 是否符合项目编码规范
3. 是否缺少边界检查
4. 是否有更好的实现方式

输出格式：列出每个问题的文件路径、行号和修复建议。

使用方式是在对话中引用模板并传参： 

用 review-pr 模板，diff 是：
git diff main...HEAD 的输出粘贴在这里
Pi 的实现差异 不同版本的 Pi 对 Skill 的加载机制有差异。请参考当前版本 docs/skills.md 中的详细说明。 如果 .pi/agent/skills/ 在你的版本里不存在，尝试放在 ~/.config/pi/skills/ 或 ~/.pi/skills/。
