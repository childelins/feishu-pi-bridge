Lesson 11 · Extension 再进阶

保存 Extension 状态 & 弹出交互 UI

时长约 8 分钟

第 10 课学了注册命令和事件钩子。现在还有两件事让你的 Extension 更完整： 持久化（把数据存下来，跨 Session 不丢）和 交互 UI（向用户弹确认框、输入框）。

本课目标 到这一课结束时，你应该能：用 pi.appendEntry 把自定义数据写入 Session 文件； 在 execute 中用 ctx.ui.confirm 和 ctx.ui.input 弹出交互框； 并判断当前运行模式（交互 vs 非交互）以优雅降级。

pi.appendEntry——持久化你的数据

默认情况下 Extension 是"无状态"的——每次启动 Pi，扩展的变量都会重置。 如果你想跨 Session 保留数据（比如计数器、配置、历史记录），用 pi.appendEntry。 

export default function (pi: ExtensionAPI) {
  // 写入一条自定义条目到 session 文件
  pi.appendEntry("counter", { value: 42 });
}

写进去的数据存储在 Session JSONL 文件中，重启 Pi 后可以通过 pi.getEntries 读取出来：

const entries = pi.getEntries("counter");
entries.forEach(e => console.log(e.data.value)); // 42
存储在哪里？ appendEntry 的数据写入到当前 Session 的 JSONL 文件（~/.pi/agent/sessions/...jsonl）。 它是持久化的但绑定在 Session 上——同一个项目的不同 Session 不会共享这些数据。 如果需要在扩展之间共享全局数据，可以用文件系统自己存（比如 ~/.pi/agent/extensions-data/）。

动手：计数器

写一个带 /count 命令的 Extension，每次调用数字加 1，并记住总数：

import type { ExtensionAPI } from "@earendil-works/pi-coding-agent";

export default function (pi: ExtensionAPI) {
  pi.registerCommand({
    name: "count",
    description: "Add one and return the current count. Stored across sessions.",
    execute: async (_ctx, args: string) => {
      const entries = pi.getEntries("counter");
      // 从已有的记录中读取最后一个值
      const lastVal = entries.length > 0
        ? entries[entries.length - 1].data.value
        : 0;
      const newVal = lastVal + 1;
      pi.appendEntry("counter", { value: newVal });
      return `Count: ${newVal}`;
    },
  });
}

动手试试

  1. 把代码保存到 ~/.pi/agent/extensions/counter.ts。
  2. 重启 Pi，输入 /count 几次，确认数字递增。
  3. 退出 Pi 重新启动，输入 /count，确认数字从上一次的值继续。

ctx.ui——给用户弹窗

在 execute 的回调中，ExecutionContext 提供了 ctx.ui 对象， 可以弹出确认框和输入框。

方法效果返回值
ctx.ui.confirm("确认执行？") 弹出是/否确认框 Promise<boolean>
ctx.ui.input("输入名字：") 弹出一个文本输入框 Promise<string>

例子：带确认的删除

pi.registerTool({
  name: "delete_file",
  description: "Delete a file with confirmation",
  parameters: {
    type: "object",
    properties: {
      path: { type: "string", description: "File path to delete" },
    },
    required: ["path"],
  },
  execute: async (ctx, params: { path: string }) => {
    const ok = await ctx.ui.confirm(`确定要删除 ${params.path} 吗？`);
    if (!ok) return { content: [{ type: "text", text: "已取消" }] };
    // 执行删除逻辑...
    return { content: [{ type: "text", text: `已删除 ${params.path}` }] };
  },
});

例子：交互式配置

pi.registerCommand({
  name: "setup",
  description: "Interactive setup",
  execute: async (ctx, _args: string) => {
    const name = await ctx.ui.input("你的名字：");
    const ok = await ctx.ui.confirm(`名字是 ${name}，确认？`);
    if (!ok) return "已取消";
    pi.appendEntry("config", { name });
    return `已保存: ${name}`;
  },
});
ctx 从哪里来 registerTool 的 execute 第一个参数就是 ExecutionContext， registerCommand 的 execute 也一样。在第 9 课里我们把它命名为 _ctx 没使用。

模式感知：交互 vs 非交互

ctx.ui 在非交互模式下不可用（比如 pi -p "..." 或 --mode json）。 如果不做判断就直接调用，Extension 会抛错或静默失败。

用 pi.mode 判断当前运行模式： 

export default function (pi: ExtensionAPI) {
  pi.registerCommand({
    name: "greet",
    description: "Say hello",
    execute: async (ctx, args: string) => {
      if (pi.mode === "interactive") {
        const name = await ctx.ui.input("你的名字：");
        return `你好，${name}！`;
      }
      // 非交互模式下直接使用参数
      const name = args.trim() || "世界";
      return `你好，${name}！`;
    },
  });
}

好的习惯是：交互模式用 UI 获取输入，非交互模式回退到命令行参数或默认值。
