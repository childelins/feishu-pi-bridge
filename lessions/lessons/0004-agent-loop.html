Lesson 4 · 核心机制

Agent Loop 详解——你按下回车后发生了什么

时长约 8 分钟

上一课你了解了 Pi 的会话生命周期。 现在往里再钻一层：按下回车后，Agent Loop 是怎么转起来的。

本课目标 到这一课结束时，你应该能说清楚：LLM 是怎么决定调用哪个工具的、工具执行后结果怎么回到 LLM、循环什么时候终止、以及出错了会发生什么。

Agent Loop 一句话定义

Agent Loop 就是： 

重复 {
  ① LLM 思考（决定下一步做什么）
  ② 如果需要 → 调用工具（Tool Call）
  ③ 执行工具，拿到结果（Tool Result）
  ④ LLM 观察结果，决定继续还是结束
} 直到 LLM 认为任务完成，输出最终回复

这个循环是 所有 AI Coding Agent 的核心。 Pi 没有魔法，只是在忠实地跑这个循环。

第一步：LLM 拿到上下文

你按下回车，Pi 先构建当前上下文（context），然后把它发给 LLM。 

发送给 LLM 的上下文包含：
├── System Prompt（系统提示词，定义 Pi 的身份和能力）
├── Context Files（AGENTS.md / CLAUDE.md 的内容）
├── 会话历史（之前的所有消息、工具调用、工具结果）
├── 当前消息（你刚输入的这句话）
└── 工具定义（当前可用的所有工具的 name / description / parameters）

System Prompt 告诉 LLM：「你是 Pi，你有这些工具可用，这是项目约定。」 工具定义告诉 LLM：「你可以调用 read、write、edit、bash，以及其他扩展注册的工具。」

工具定义是动态生成的 每次 Agent Loop 开始时，Pi 会根据当前已加载的 Extension 生成工具定义列表。如果你的 Extension 注册了新工具，LLM 下一轮就能看到它。

第二步：LLM 决定行动

LLM 拿到完整上下文后，生成一个回复。这个回复可能是：

LLM 回复类型含义例子
文本内容 直接回复用户，不需要调用工具 "这个项目的入口文件是 src/index.ts"
工具调用（Tool Call） 需要执行一个工具来获取信息 read(path="src/index.ts")
多个工具调用 需要并行执行多个工具（如果互不依赖） 同时 read 三个文件

LLM 通过 Function Calling（函数调用）机制来触发工具调用。 它输出的不是普通文本，而是一个结构化的 JSON 对象： 

{
  "tool": "read",
  "arguments": { "path": "src/index.ts" }
}

第三步：Pi 执行工具

LLM 发出工具调用后，不是 LLM 自己去执行—— 是 Pi 的运行时（AgentSession）负责执行工具，并把结果放回上下文。 

LLM 输出 tool call
        ↓
Pi 解析 tool name + arguments
        ↓
查找对应的工具执行函数（内置或 Extension 注册）
        ↓
执行工具（如读取文件、运行 shell 命令）
        ↓
捕获结果（输出内容 / 错误信息 / 退出码）
        ↓
将结果封装为 ToolResultMessage，追加到会话历史
        ↓
回到第一步：把更新后的上下文再次发给 LLM

这个「执行 → 追加结果 → 再次发给 LLM」的过程，就是 Agent Loop 的一轮。

第四步：LLM 观察结果，决定继续还是结束

工具执行结果回到 LLM 后，LLM 读取它，然后决定：

  • 任务完成了 → 输出最终文本回复给用户，Loop 结束，回到 >> 提示符等待下一次输入。
  • 还需要更多信息 → 再次发出工具调用，Loop 继续。
  • 出错了 → LLM 看到错误信息，尝试修正后重新调用工具（自动重试）。

自动重试：如果工具执行失败（如文件路径不存在、shell 命令报错），Pi 会把错误信息放回给 LLM，LLM 会尝试修正参数后重新调用。这个过程最多重试若干次，超过上限则终止并报告错误。

一个完整循环示例

假设你输入：「帮我看看 src/ 目录下有哪些 .ts 文件，然后读前两个文件的开头。」 

第 1 轮：
  你：「帮我看看 src/ 目录下有哪些 .ts 文件...」
  LLM：「我需要先列出文件」→ tool call: bash("find src/ -name '*.ts'")
  Pi 执行 → 结果：["src/a.ts", "src/b.ts", "src/c.ts"]
  LLM 观察结果...

第 2 轮：
  LLM：「好，我看到了三个文件，让我读前两个」→ tool call: read("src/a.ts") + read("src/b.ts")（并行）
  Pi 执行 → 结果：a.ts 的内容 / b.ts 的内容
  LLM 观察结果...

第 3 轮：
  LLM：「我已经读完了，这是前两个文件的内容...」→ 文本回复
  Loop 结束，回到 >> 提示符。

思考（Thinking）是什么？

某些模型（如 Claude）支持 Thinking / Reasoning 模式。 在这个模式下，LLM 在输出正式回复之前，会先输出一段「内部思考」—— 这段思考 不展示给用户（除非你开了 verbose）， 但会被用来指导后续的工具调用。

你可以在启动时通过 --thinking 参数控制思考等级： off、minimal、low、medium、high、xhigh。

思考等级越高 LLM 用来「想」的 token 越多，通常结果越准确，但速度越慢、成本越高。根据任务复杂度选择合适的等级。

Loop 终止条件

Agent Loop 在以下几种情况下终止：

终止原因说明
LLM 输出文本回复任务完成，LLM 认为不需要再调用工具
上下文溢出会话历史太长，触发自动 Compaction（压缩）
超过最大轮次防止无限循环，Pi 有内置的安全上限
用户中断按 Escape 或 Ctrl+C，终止当前 Loop
工具执行出错且无法自动恢复重试次数耗尽，Loop 终止并报错
